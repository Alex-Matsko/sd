# ТЗ: Омниканальная сервис-деск система для MSP «Открытые Горизонты»

Версия 1.0 · Июль 2026 · Документ предназначен для разработки силами ИИ-агента (Claude Code)

---

## 1. Общие сведения

**Заказчик:** ООО «Открытые Горизонты» (o-horizons.com) — MSP-компания: аутсорсинговое сопровождение серверов и пользователей, внедрение сервисов, аренда ресурсов, поддержка 1С.

**Цель:** собственная омниканальная сервис-деск система, спроектированная под специфику MSP (multi-tenant, договоры, тарифные SLA, биллинг времени), в соответствии с практиками ITIL 4. Система разрабатывается с нуля и эксплуатируется параллельно с существующим ServiceDesk Plus MSP до полного перехода. Миграция исторических данных из SDP в объём первой версии не входит.

**Масштаб на старте:** 5 инженеров, ~50 заявок/день. Архитектура должна допускать рост до 20+ инженеров и 500+ заявок/день без переписывания ядра.

**Хостинг:** инфраструктура заказчика — Docker Compose на отдельной ВМ (Linux). Никаких внешних SaaS-зависимостей, кроме API мессенджеров и телефонии.

## 2. Каналы приёма заявок

Первая очередь: Telegram, MAX, Email, телефония Plusofon, веб-портал клиента. WhatsApp — в архитектуре предусмотреть как будущий канал (интерфейс канала должен быть абстрагирован), но не реализовывать.

Требования к омниканальности:

1. **Единая карточка заявки** независимо от канала. Канал — атрибут заявки и каждого сообщения в ней.
2. **Ответ уходит в канал обращения.** Инженер пишет в интерфейсе системы; клиент получает ответ в TG/MAX/почте, откуда писал.
3. **Склейка диалога:** последующие сообщения клиента по открытой заявке прикрепляются к ней, а не создают новую. Правило: одна открытая заявка — сообщение уходит в неё; несколько открытых — бот в мессенджере показывает кнопки со списком заявок (ID + тема + статус) и просит выбрать, к какой относится сообщение, либо «Создать новую»; открытых нет — создаётся новая. В email при нескольких открытых привязка по In-Reply-To/тегу [#ID]; если их нет — к последней активной заявке с пометкой для диспетчера. Клиент в мессенджере может явно создать новую заявку командой/кнопкой.
4. **Автоидентификация клиента:** по Telegram user ID, MAX ID, email-адресу, номеру телефона (входящий звонок Plusofon). Идентификаторы хранятся в карточке контакта. Неопознанное обращение попадает в очередь «Неизвестные» — диспетчер привязывает контакт к организации, идентификатор сохраняется для последующих обращений.
5. **Вложения** (фото, документы, скриншоты) принимаются из всех каналов и сохраняются в заявке.

### 2.1 Telegram
Бот на Telegram Bot API (long polling или webhook — на усмотрение исполнителя, но long polling проще при self-hosted без белого IP). Функции: создание заявки, диалог по заявке, кнопка «Мои заявки» — список открытых заявок со статусами и выбор активной для переписки (последующие сообщения идут в выбранную, пока клиент не переключится или бот не переспросит по правилу склейки), просмотр статусов, оценка после закрытия (1–5).

### 2.2 MAX
Аналогично Telegram. У заказчика есть работающий опыт интеграции с MAX (long polling bridge на Python + SQLite mapping) — использовать как референс, код может быть предоставлен.

### 2.3 Email
Приём: IMAP-опрос ящика поддержки (интервал ≤ 60 сек). Идентификация клиента: сначала по точному email-адресу контакта (адресов может быть несколько, включая личные на публичных доменах — gmail, yandex и т.п.); если адрес неизвестен, но его домен зарегистрирован в карточке организации как корпоративный — письмо автоматически привязывается к этой организации, а контакт создаётся и попадает диспетчеру на подтверждение. Публичные домены (справочник в системе: gmail.com, yandex.ru, mail.ru и т.д.) из автопривязки по домену исключены — такие письма идут в очередь неопознанных. Парсинг: тема + тело → заявка; ответы клиента по In-Reply-To/References и тегу [#ID] в теме → в существующую заявку. Отправка: SMTP, уведомления и ответы инженера с тегом заявки в теме. Вложения сохраняются.

### 2.4 Телефония Plusofon
Интеграция через API/webhooks Plusofon (актуальную документацию API исполнитель изучает на этапе реализации):

- Входящий звонок → всплывающее событие у диспетчера: карточка клиента (если номер опознан) + кнопка «Создать заявку» с предзаполненным контактом.
- Если звонок пропущен — автоматически заявка «Пропущенный звонок» в очередь диспетчера.
- В карточке заявки, созданной по звонку: номер, время, длительность, ссылка/файл записи разговора, если Plusofon её отдаёт.
- Исходящий звонок из карточки заявки (click-to-call через API) — желательно, не блокирует релиз.

### 2.5 Веб-портал клиента
См. раздел 7.

## 3. Модель данных

Ядро (multi-tenant для MSP):

- **Organization** — клиент-организация: реквизиты, статус, ответственный менеджер, список корпоративных email-доменов (для автопривязки писем).
- **Contact** — сотрудник клиента: ФИО, должность, email-адреса (несколько, в т.ч. личные на публичных доменах), телефоны, TG ID, MAX ID, признак «VIP», право видеть все заявки организации или только свои.
- **Contract** — договор организации: номер, срок, тариф, включённые часы в месяц, ставка за час сверх лимита, статус.
- **Tariff / SLA Policy** — тарифный план, определяет SLA-матрицу (время реакции и решения по приоритетам) и бизнес-календарь (напр. 5×8 или 8×20 или 24×7). Тарифов несколько, у каждой организации — один через договор.
- **Asset (CMDB, минимальный)** — актив клиента: тип (сервер, рабочая станция, база 1С, арендуемый ресурс, сетевое устройство), имя, описание, привязка к организации. Заявка может ссылаться на актив. Полноценный CMDB с discovery — вне объёма v1.
- **Ticket** — заявка: тип (Incident / Service Request), организация, контакт, договор (подтягивается автоматически), актив (опционально), канал, тема, приоритет, статус, исполнитель, группа, SLA-дедлайны, метки.
- **Message** — сообщение в заявке: направление (входящее/исходящее/внутренняя заметка), канал, автор, текст, вложения. Внутренние заметки клиенту не видны и не отправляются.
- **TimeEntry** — списание времени: заявка, инженер, длительность, дата, комментарий, признак «биллингуемое».
- **User** — сотрудник компании: роль, группы, каналы уведомлений.

## 4. Процессы (ITIL 4, прагматичный объём)

Реализуются **Incident Management** и **Request Fulfilment**. Problem Management — только как метка «повторяющаяся проблема» и связь заявок между собой (parent/child), без отдельного процесса. Change Management — вне объёма v1, в модели данных не закладывать, но не блокировать расширение.

### 4.1 Классификация и приоритет
Тип заявки: Incident (что-то сломалось) / Service Request (запрос услуги). Категория: двухуровневый настраиваемый справочник (например: Серверы → Windows, 1С → Обновление конфигурации, Пользователи → Рабочее место, Аренда ресурсов → …).

Приоритет вычисляется по матрице Impact × Urgency (настраиваемая, 3×3, результат P1–P4), диспетчер/инженер может переопределить вручную с указанием причины (фиксируется в истории).

### 4.2 Статусная модель
`Новая → Назначена → В работе → Ожидает клиента → Ожидает третью сторону → Решена → Закрыта` (+ `Отменена`). Правила:

- «Ожидает клиента» и «Ожидает третью сторону» ставят SLA-таймер решения на паузу.
- Ответ клиента в заявке со статусом «Ожидает клиента» автоматически возвращает её «В работе» и уведомляет исполнителя.
- «Решена» → «Закрыта»: автозакрытие через N дней (настройка, по умолчанию 3) без ответа клиента; клиент может переоткрыть ответом в течение этого срока.
- При закрытии клиенту отправляется запрос оценки (1–5) в его канал.

### 4.3 SLA
SLA-политика привязана к тарифу договора. Для каждого приоритета: время реакции (первый ответ инженера) и время решения. Таймеры считаются по бизнес-календарю тарифа (рабочие часы, праздники РФ — справочник в системе, редактируемый).

Эскалации: при достижении 75% SLA — уведомление исполнителю; при 100% — уведомление руководителю группы и пометка нарушения. Все нарушения фиксируются для отчётности (по каждой заявке видно: SLA выполнен/нарушен, по реакции и по решению отдельно).

### 4.4 Маршрутизация и назначение
Правила автоназначения (настраиваемые, порядок применения по приоритету правила): по категории → группа; по организации → выделенный инженер; иначе — общая очередь диспетчера. Внутри группы — вручную диспетчером или самоназначение инженером. Балансировку по загрузке в v1 не делать.

## 5. Учёт времени и биллинг

- Инженер списывает время на заявку: таймер старт/стоп в карточке + ручной ввод. Каждая запись — биллингуемая или нет.
- Договор содержит включённые часы/месяц и ставку сверх лимита. Система в реальном времени показывает по каждой организации: израсходовано из пакета за текущий месяц, перерасход в часах и деньгах.
- Месячный отчёт по организации: список заявок, списанное время (billable/non-billable), расход пакета, сумма сверх пакета. Экспорт в PDF и XLSX. Интеграция с 1С:УНФ — вне объёма v1 (целевая для v2), но все биллинговые данные должны быть доступны через API.

## 6. Роли и права

- **Инженер:** свои и групповые заявки, списание времени, база знаний.
- **Диспетчер:** все заявки, очередь неопознанных, назначение, создание заявок по звонкам, управление контактами.
- **Руководитель:** всё выше + отчёты, настройка SLA/тарифов/правил, управление пользователями.
- **Администратор:** полный доступ, системные настройки.
- **Клиент (портал):** свои заявки; если флаг у контакта — все заявки своей организации; расход пакета часов по договору.

## 7. Портал клиента

Отдельный веб-интерфейс (поддомен): вход по email + пароль или magic-link на почту. Функции: создать заявку (форма с категорией и вложениями), список и просмотр своих заявок, переписка, статус расхода часов по договору (если включено для организации), оценка закрытых заявок. Мобильная вёрстка обязательна.

## 8. Интерфейс инженера/диспетчера

Веб-SPA. Ключевые экраны:

1. **Очередь заявок** — табличный список с фильтрами (статус, организация, исполнитель, приоритет, канал, SLA-риск), сохраняемые фильтры, счётчики. Цветовая индикация SLA (зелёный/жёлтый/красный).
2. **Карточка заявки** — переписка в виде треда (визуально различать канал и внутренние заметки), панель свойств, таймер времени, история изменений (audit log), связанные заявки, быстрые ответы-шаблоны.
3. **Дашборд** — открытые по статусам, SLA-риски сегодня, загрузка инженеров, поток за неделю.
4. **Организации/контакты/договоры/активы** — CRUD-справочники.
5. **Отчёты** — раздел 9.
6. **Настройки** — SLA/тарифы, категории, правила маршрутизации, шаблоны ответов, каналы, календари.

Уведомления инженерам: в интерфейсе (websocket) + в Telegram (личный бот-канал системы, инженер привязывает свой TG в профиле).

Интерфейс полностью на русском языке. Дизайн — сдержанный, рабочий, без визуального шума.

## 9. Отчётность

Все отчёты — фильтр по периоду и организации, экспорт XLSX/PDF:

1. SLA-отчёт: % выполнения по реакции и решению, разрез по организациям и приоритетам, список нарушений.
2. Отчёт клиенту за период: заявки, время, расход пакета (форма пригодная для отправки клиенту).
3. Загрузка инженеров: списанное время, количество решённых, среднее время решения.
4. Поток заявок: по каналам, категориям, организациям, динамика.
5. CSAT: средняя оценка, разрез по инженерам и организациям.

## 10. Технические требования

**Стек (предложение исполнителю, допускается обоснованная замена):**

- Backend: Python 3.12+, FastAPI, SQLAlchemy, Alembic (миграции), Celery или встроенный планировщик задач для SLA-таймеров/автозакрытий/IMAP-опроса.
- БД: PostgreSQL 16. Кэш/очереди/websocket pub-sub: Redis.
- Frontend: React + TypeScript (Vite), SPA для сотрудников + отдельная сборка портала клиента.
- Файлы-вложения: локальное хранилище на диске ВМ (том Docker), структура по годам/месяцам; лимит размера вложения — настройка.
- Развёртывание: Docker Compose (app, worker, db, redis, nginx). Один файл `.env` с конфигурацией. Обратный прокси nginx с TLS (сертификаты заказчика/Let's Encrypt).

**Нефункциональные требования:**

- Данные не покидают инфраструктуру заказчика (кроме обмена с API мессенджеров/Plusofon).
- Бэкапы: pg_dump по расписанию + вложения; скрипт бэкапа в комплекте, восстановление документировано и проверено.
- Audit log всех изменений заявок и настроек (кто, когда, что).
- Аутентификация сотрудников: логин/пароль, хранение паролей bcrypt/argon2, сессии/JWT. 2FA (TOTP) — желательно.
- REST API с OpenAPI-документацией для всех сущностей — под будущие интеграции (1С, мониторинг, WhatsApp).
- Логи приложения структурированные (JSON), ротация.
- Тесты: покрытие бизнес-логики SLA-расчётов и маршрутизации обязательно (это самая рискованная логика).

## 11. Этапы разработки (итерации для ИИ-агента)

Каждый этап завершается работающим, разворачиваемым результатом и коротким демо-сценарием проверки.

1. **Ядро:** модель данных, миграции, REST API, аутентификация, CRUD организаций/контактов/договоров/тарифов, заявки со статусной моделью, Docker Compose.
2. **Интерфейс сотрудника:** очередь, карточка заявки, справочники, назначение, внутренние заметки.
3. **SLA-движок:** матрица приоритетов, календари, таймеры, паузы, эскалации, индикация. Тесты.
4. **Канал Email** (полный цикл вход/выход) + уведомления инженеров в Telegram.
5. **Каналы Telegram и MAX** для клиентов: идентификация, диалог, вложения, очередь неопознанных.
6. **Plusofon:** события звонков, заявки по пропущенным, карточка звонка.
7. **Время и биллинг:** таймеры, пакеты часов, перерасход, отчёт клиенту.
8. **Портал клиента.**
9. **Отчёты, дашборд, CSAT, автозакрытие, шаблоны ответов.**
10. **Стабилизация:** бэкапы, документация по развёртыванию и администрированию, нагрузочная проверка (500 заявок/день).

## 12. Критерии приёмки (сквозной сценарий)

1. Клиент пишет в Telegram → система опознаёт его по TG ID → создаётся заявка с правильными организацией, договором и SLA по тарифу.
2. Инженер отвечает из карточки → клиент получает ответ в Telegram; SLA реакции зафиксирован.
3. Инженер ставит «Ожидает клиента» → таймер решения на паузе; клиент отвечает → заявка «В работе», инженер уведомлён.
4. Инженер списывает 1.5 часа (billable) → расход пакета организации увеличился; при исчерпании пакета часы считаются по ставке сверх лимита.
5. Заявка решена → через 3 дня автозакрытие → клиенту ушла просьба об оценке → оценка видна в отчёте CSAT.
6. Пропущенный звонок с известного номера → заявка «Пропущенный звонок» с карточкой клиента в очереди диспетчера.
7. Письмо на ящик поддержки с вложением → заявка с вложением; ответ инженера ушёл письмом; ответ клиента на письмо попал в ту же заявку.
8. SLA-отчёт за месяц показывает нарушения и % выполнения, совпадающие с фактическими данными заявок.
9. Восстановление из бэкапа на чистой ВМ по документации — работоспособная система с данными.

## 13. Вне объёма v1 (зафиксировано)

WhatsApp, Change Management, полноценный Problem Management, миграция из ServiceDesk Plus, discovery активов, балансировка нагрузки при автоназначении, мобильные приложения.

**Интеграция с 1С:УНФ** — целевая для v2: передача в УНФ данных для выставления счетов и учёта (организация, договор, период, заявки, часы billable, перерасход пакета в часах и деньгах). Требование к v1: REST API обязан отдавать эти данные в машиночитаемом виде за произвольный период, чтобы интеграция не потребовала доработок ядра.
