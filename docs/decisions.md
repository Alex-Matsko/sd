# Решения и уточнения к ТЗ

Дополняет [`TZ.md`](./TZ.md). Каждая запись — решение, принятое по ходу разработки там, где исходное ТЗ оставляло свободу трактовки. Новые решения дописывать в конец, не переписывать историю.

## Зафиксировано с заказчиком (2026-07-20)

### Номер заявки (Ticket ID)
Формат: `OH-<порядковый номер>`, например `OH-1042`. Сквозная нумерация по всей системе, без сброса по годам. Используется как тег `[#OH-1042]` в теме письма, в кнопках Telegram/MAX, в портале и в API.

### SLA/договор при отсутствии активного договора у организации
При создании заявки, если у организации нет активного договора: применяется тариф по умолчанию (базовый SLA), заявка создаётся и обрабатывается сразу. В карточке — пометка «нет активного договора» для диспетчера/руководителя, организационный вопрос решается параллельно, не блокируя работу по заявке.

### Биллинг на границе пакета часов
Если списание времени пересекает остаток пакета (например, остался 1 час, списано 3), запись делится пропорционально: часть — в счёт пакета, часть — по ставке сверх лимита. На уровне модели `TimeEntry` одна пользовательская запись времени может породить два расчётных отрезка (пакетный/сверхлимитный) для биллинга и отчётности, при этом пользователь видит и редактирует её как одну запись.

### SLA при переоткрытии заявки клиентом
Клиент отвечает на «Решённую» заявку в течение окна автозакрытия → заявка возвращается «В работе», исполнитель уведомлён. SLA по решению **не пересчитывается**: исходный результат (выполнено/нарушено) фиксируется как окончательный на момент первого перевода в «Решена». Работа после переоткрытия не создаёт новый дедлайн решения. Факт переоткрытия фиксируется в истории заявки (audit log) как есть, отдельная метрика reopen-rate — не в объёме v1, но поле для неё не блокируется на будущее.

## Предположения по умолчанию (озвучены, возражений не поступило — считаются принятыми)

- **Матрица Impact × Urgency**: стандартная ITIL-подобная таблица 3×3 → P1–P4, задаётся сид-данными, редактируется в настройках.
- **Порядок правил маршрутизации**: правило «организация → выделенный инженер» (более специфичное) применяется раньше правила «категория → группа»; при отсутствии совпадений — общая очередь диспетчера. Порядок — настраиваемое поле, не хардкод.
- **Бизнес-календари**: универсальная структура (окна по дням недели + справочник праздников РФ, редактируемый по годам). Сид-данные: 2–3 примера календаря (обычный офисный, продлённый, 24×7) с рабочими часами-заглушками — реальные часы для тарифов заказчик задаёт через настройки.
- **Сущность «Группа/команда» (Team)**: не описана явно в разделе 3 ТЗ, но нужна для маршрутизации по категории и для эскалации «руководителю группы». Вводится как отдельная сущность с полем «руководитель группы».
- **VIP-признак контакта**: в v1 — только визуальная метка для диспетчера/инженера, без автоматического влияния на приоритет или SLA. Поведение можно добавить позже без изменения схемы (поле уже булево).
- **Непривязанный контакт при автопривязке письма по домену организации**: заявка создаётся и обрабатывается сразу; контакт помечается «не подтверждён» до подтверждения диспетчером — не блокирует обработку заявки.
- **Мессенджер: клиент не ответил на запрос бота «к какой заявке относится сообщение»**, а написал следующее сообщение: ведём себя как в email-правиле — прикрепляем к последней активной заявке с пометкой для диспетчера.
- **Лимит размера вложения по умолчанию**: 25 МБ, настраиваемый параметр.

## Открытые вопросы (не блокируют Этап 1, решить до соответствующего этапа)

- Точные рабочие часы бизнес-календарей по факту (нужны реальные значения по тарифам заказчика) — до Этапа 3.
- Домен/поддомен и способ выпуска TLS-сертификата для портала клиента — до Этапа 8.
- Учётные данные интеграций: Telegram bot token, MAX bridge, Plusofon API-ключи, IMAP/SMTP почтового ящика поддержки — по мере входа в соответствующие этапы (4–6).

## Технические примечания по реализации Этапа 3 — SLA-движок (2026-07-22)

### Модель хранения дедлайнов
Срок решения хранится как абсолютный `sla_resolution_due_at`, а не как «оставшийся бюджет» — при постановке на паузу и снятии с паузы пересчитывается как `created_at + рабочее_время(бюджет_SLA + накопленная_пауза_в_рабочих_минутах)`. Пауза считается в рабочих минутах календаря (не в календарных), чтобы пауза на выходные не съедала рабочий бюджет. Один и тот же механизм пересчёта используется и при снятии с паузы, и при смене приоритета вручную.

### Фиксация результата (met/breached)
`sla_reaction_met` и `sla_resolution_met` — тройное состояние (`NULL` = ещё не решено, `true`/`false` = зафиксировано). Фиксируются один раз: по реакции — первым исходящим сообщением инженера или сработавшей эскалацией 100%; по решению — первым переводом в «Решена» или эскалацией 100%. После фиксации значение не пересчитывается (согласуется с более ранним решением про SLA при переоткрытии). Смена приоритета обнуляет `met=false`, если новый дедлайн ещё не наступил (иначе нарушение, уже случившееся по старому таймеру, осталось бы «зависшим» несправедливо), но никогда не трогает уже свершившийся положительный результат.

### Эскалации 75%/100%
Отдельный процесс-воркер (`app/worker.py`, контейнер `worker` в compose) раз в `WORKER_INTERVAL_SECONDS` (по умолчанию 60 с) сканирует открытые заявки без зафиксированного результата. На 75% бюджета — уведомление исполнителю (или руководителю группы, если не назначен); на 100% — фиксация нарушения и уведомление исполнителю **и** руководителю группы одновременно. Разовые метки `sla_*_warned_at`/`sla_*_escalated_at` на заявке гарантируют, что каждое уведомление уходит ровно один раз; сбрасываются при пересчёте по смене приоритета.

### Уведомления — только in-app на этом этапе
Добавлена таблица `notifications` и раздел в topbar (колокольчик, список, «прочитать все»). Доставка в личный Telegram инженера — по ТЗ раздел 8, реализуется в Этапе 4 вместе с остальной телеграм-интеграцией; модель уведомлений уже не потребует миграции под это.

### Обратная совместимость: заявки, созданные до Этапа 3
Колонки SLA существовали как nullable ещё с Этапа 1 (расчёт на будущее движка), поэтому заявки, созданные в Этапах 1–2, имеют `sla_reaction_due_at`/`sla_resolution_due_at = NULL`. При каждом старте `app.seed` теперь докатывает дедлайны для таких заявок — но **только если заявка ещё не продвинулась** (нет ни первого ответа, ни решения): для уже отвеченных/решённых до появления движка результат не переоценивается задним числом (не «придумываем» нарушение/выполнение по правилу, которое в момент события не действовало). Такие заявки остаются в состоянии «без SLA» — по замыслу, это не баг, а сознательная граница ответственности движка.

### Гонка контейнеров при старте
`worker` и `nginx` теперь зависят от `app` через Docker healthcheck (`condition: service_healthy`), а не просто от готовности `db` — иначе воркер иногда успевал обратиться к БД до применения миграций `app`-контейнером (замечено в разработке: первый цикл воркера падал на `UndefinedColumn`, второй проходил штатно благодаря ретраю, но само по себе полагаться на ретрай — не решение).

## Технические примечания по реализации Этапа 4 — Email-канал (2026-07-22)

### Универсальные настройки каналов вместо `.env`
По запросу заказчика учётные данные интеграций (IMAP/SMTP сейчас; Telegram bot token, MAX bridge, Plusofon API-ключи на будущих этапах) вынесены из `.env` в БД и редактируются в интерфейсе (Настройки → Каналы, раздел 8 ТЗ). Реализовано одной универсальной таблицей `integration_settings` (канал → JSON `config` + JSON `secrets_encrypted`), а не отдельной типизированной моделью на каждый канал — новый канал не требует новой миграции, только согласование ключей конфига на уровне приложения. Секреты шифруются `Fernet` (ключ `SETTINGS_ENCRYPTION_KEY` в `.env` — это единственное, что осталось «инфраструктурным» и не переехало в БД, т.к. без него сами секреты не расшифровать) и никогда не отдаются через API в открытом виде — только флаг, какие ключи заданы. Frontend показывает пароль/токен как пустое поле «оставьте пустым, чтобы не менять» плюс кнопку «Очистить», а не сам секрет.

### Приоритет привязки входящего письма к заявке
Реализовано строго по разделу 2.3: (1) заголовки In-Reply-To/References против `Message.email_message_id`, сохранённого на предыдущих исходящих письмах этой же заявки; (2) тег `[#OH-<номер>]` в теме; (3) единственная открытая заявка контакта; (4) несколько открытых заявок без совпадения по заголовкам/тегу → последняя активная (по `updated_at`, с `id` как детерминированным тай-брейкером — `now()` в Postgres одинаков для всех операторов одной транзакции, поэтому только timestamp недостаточен) с автоматической внутренней заметкой-пометкой для диспетчера; (5) открытых заявок нет → новая заявка. Новая заявка от письма создаётся с типом Incident и приоритетом P3 (ручной, с причиной «автоматически создано по email») — реальные Impact/Urgency клиент по почте не указывает, дальнейшую классификацию делает диспетчер/инженер.

### Очередь «Неизвестные»
Письма с адреса, не совпадающего ни с одним контактом и не относящегося к зарегистрированному корпоративному домену организации (включая все публичные домены из `PublicEmailDomain`), попадают в заявки организации-заглушки «Неизвестные» (создаётся лениво при первом обращении, ищется по имени — тот же приём, что и зарезервированный `code` тарифа по умолчанию). Контакт создаётся с `is_confirmed=false`. Полноценный UI «перепривязать контакт и его заявки к другой организации» не реализован в этом этапе — диспетчер видит такие заявки через фильтр по организации «Неизвестные» в общей очереди; отдельный экран разбора — кандидат на один из следующих этапов, если появится такая необходимость.

### Обработка ошибок при IMAP-опросе
Письмо считается обработанным, как только оно прочитано через `FETCH RFC822` (IMAP помечает `\Seen` автоматически) — даже если разбор/создание заявки после этого упало с исключением. Альтернатива («не помечать как прочитанное при ошибке») ведёт к бесконечному повтору одного и того же письма при системной ошибке разбора и риску дублирования уже частично созданных сущностей. Каждая ошибка полностью логируется для последующего ручного разбора; это компромисс, осознанно принятый для v1.

### Что не реализовано в этом проходе Этапа 4
ТЗ относит к Этапу 4 также «уведомления инженерам в Telegram» — это осталось не сделанным (нет Telegram-бота, нет доставки уведомлений о новых заявках/эскалациях в личный чат инженера). Инфраструктура настроек уже готова к этому: в `integration_settings` заведена заглушка канала `telegram` (переключатель «включён», без полей конфигурации), форма появится вместе с реализацией самого бота.

## Технические примечания по реализации Этапа 5 — канал MAX (2026-07-22)

### Референсной реализации не было — разработка с нуля
Ранее упомянутый в ТЗ рабочий MAX-бридж заказчика (long polling + SQLite mapping) в итоге так и не был доведён до конца и не был предоставлен как референс. Канал реализован с нуля по официальной документации `dev.max.ru/docs-api`, полученной через веб-поиск на момент разработки (2026-07-22) — **без доступа к боевому токену бота для проверки реальных ответов сервера**. Зафиксированные из документации факты:

- Домен API: `platform-api2.max.ru` (старый `platform-api.max.ru` выводится из эксплуатации к 2026-07-19 — уже пройдено на момент разработки, поэтому используется только новый).
- Авторизация: заголовок `Authorization: <токен>` без префикса `Bearer`.
- Приём: `GET /updates` (long polling, поля `marker`/`timeout`/`limit`), продакшен официально рекомендует webhook, но клиент прямо просил long polling (как и было в его прежнем бридже) — самостоятельный TLS-эндпоинт с публичным сертификатом не нужен, что соответствует общему принципу ТЗ «self-hosted без белого IP».
- Отправка: `POST /messages` (`chat_id`, `text`, вложения через `attachments`, кнопки — через `inline_keyboard`).
- Структура события: `update_type` (`message_created`/`message_callback`/`bot_started`/…), сообщение — `sender.user_id`, `recipient.chat_id`, `body.text`.

Разбор входящих событий (`_extract_incoming` в `app/services/max_channel.py`) сознательно защитный — все обращения к полям через `.get()` с проверкой на `None`, а не через прямой доступ по ключу: если реальный сервер вернёт другую форму данных, обработчик залогирует и пропустит событие вместо падения. **Перед вводом в эксплуатацию обязательно нужно завести тестового бота и сверить реальные структуры событий с кодом** — вся точка входа для правок сосредоточена в этой одной функции.

### Маркер long polling хранится в БД, не в памяти процесса
`marker` (курсор `/updates`) сохраняется в `integration_settings.config._marker` после каждого опроса — иначе перезапуск воркера заново получал бы уже обработанные события. Побочный эффект: если администратор пересохранит настройки канала MAX через интерфейс, маркер будет затёрт (форма отправляет весь `config` целиком) и бот один раз перечитает недавние события повторно. Это осознанно принятый компромисс (редкое действие администратора, ограниченный и не разрушительный повтор) в обмен на простоту — не заводили отдельное хранилище только ради этого поля.

### MAX-опрос — в отдельном потоке воркера, не в общем тике
`GET /updates` блокируется на сервере до `timeout` секунд (до 90). Если бы этот вызов стоял в общем цикле воркера рядом с SLA-эскалациями и опросом почты (`app/worker.py`), долгий опрос MAX откладывал бы SLA-эскалации на то же время. Поэтому MAX-поллинг вынесен в отдельный `threading.Thread`, независимый от основного тика; общий цикл (SLA раз в минуту + email по своему интервалу) не блокируется.

### Правила склейки диалога и очередь «Неизвестные» — идентичны email
Одна открытая заявка — сообщение в неё; несколько — бот показывает кнопки списком (плюс «Создать новую»), при ответе обычным текстом вместо кнопки — привязка к последней активной с внутренней пометкой для диспетчера (тот же принцип, что и в письмах, см. примечания Этапа 4). Активная заявка контакта хранится в новой таблице `channel_conversation_states` (одна запись на пару канал+внешний ID пользователя) и держится, пока клиент не переключится через «Мои заявки» или заявка не перестанет быть открытой. Явное создание новой заявки — командой/кнопкой «Создать новую» — переводит диалог в режим ожидания следующего сообщения как описания проблемы. Неопознанные отправители (MAX ID не привязан ни к одному контакту) уходят в ту же организацию-заглушку «Неизвестные», что и email (`app/services/unknown_queue.py`, вынесен в общий модуль вместо дублирования).

### CSAT: добавлено раньше срока (Этап 9 по плану), т.к. это часть функций бота
Раздел 2.1/2.2 ТЗ прямо перечисляет «оценка после закрытия (1–5)» как функцию бота — поэтому запрос оценки (кнопки 1–5) и приём ответа реализованы уже сейчас: при переводе заявки в «Закрыта» (`services/tickets.transition_status`) для заявок канала MAX отправляется запрос, ответ по кнопке фиксируется в `Ticket.csat_rating`/`csat_rated_at` (первая оценка финальна, повторные попытки игнорируются). Полноценная отчётность CSAT (сводные показатели, разрез по инженерам/организациям) остаётся в Этапе 9 — колонки уже есть, повторной миграции не потребуется.

### Вложения от клиента через MAX — только при наличии прямой ссылки
MAX отдаёт вложения как типизированные payload'ы; официально описанный способ получения файла по токену через отдельный запрос не проверялся вживую. Вложение сохраняется, только если в его payload сразу есть `url` для скачивания (по документации это в целом относится к изображениям) — иначе событие логируется и пропускается с пометкой «требует проверки на боевом токене», без попытки угадать неподтверждённый API вызова. Исходящие вложения (файл от инженера клиенту в MAX) в этом проходе не реализованы вовсе.
